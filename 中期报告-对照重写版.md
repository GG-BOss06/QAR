## 二、项目背景与意义

### 2.1 研究背景

#### 2.1.1 航空运维数据的安全挑战

QAR（Quick Access Recorder，快速访问记录器）数据是航空运行安全监控、飞行品质分析、故障定位、运行复盘和长期运行评估中的关键基础数据。与普通业务文件相比，QAR 数据具有更强的专业性和敏感性，通常涉及飞行高度、速度、姿态、发动机工况、控制面偏转以及其他飞行参数信息。这类数据不仅与航空运行安全直接相关，而且在事故调查、训练评估、运行质量提升等工作中具有长期使用价值。

航空运行数据的一个显著特点是“保存周期长”。很多数据需要在较长时间范围内进行留存和追溯，因此它面临的安全问题并不仅仅是“当前是否会泄露”，还包括“未来是否可能被回溯破解”。与此同时，QAR 数据并非只由一个主体使用，而是在航司、运行控制部门、维修单位、飞行品质监控岗位等不同主体之间流动和共享。如何在保证安全的前提下实现可控共享，成为该类系统设计中的核心问题。

传统文件管理系统往往采用“明文存储 + 账号权限控制”的方式，虽然便于开发和部署，但在高敏感场景中存在明显不足。一方面，一旦服务器被直接读取，明文文件会面临整体泄露风险；另一方面，传统 RBAC 角色模型虽然可以表达“管理员”和“普通用户”的区别，但很难细化到“某航司中的某类岗位在某类业务场景下访问特定数据”这种更复杂的共享需求。因此，航空运维数据场景天然需要一种兼顾高强度加密保护和细粒度共享控制的技术路线。

#### 2.1.2 量子计算对传统密码体系的冲击

当前广泛应用的公钥密码体系，如 RSA、ECC 等，主要建立在整数分解、离散对数等经典数学难题之上。在经典计算模型下，这些机制依然具有较好的安全性，但在量子计算背景下，其长期可靠性正面临新的挑战。尤其是基于 Shor 算法的量子攻击模型，被普遍认为会对传统公钥密码体系构成根本性冲击。

对于航空运行数据这种高敏感、长期留存的数据资产而言，量子风险并不只是一个远期理论问题。因为只要攻击者今天先截获加密数据，即使短期内无法破解，也可能在未来量子计算能力成熟后再进行解密，这种“先截获、后破解”的风险对于长期存档数据尤其突出。因此，面向未来开展后量子安全架构研究，并不是脱离现实的前瞻性想象，而是与航空数据长期保密需求直接相关的现实问题。

近年来，国际上已经明显进入后量子密码迁移阶段。NIST 已经发布首批后量子密码标准，全球范围内围绕后量子算法的标准化、迁移和应用验证工作也在加快推进。从这一背景出发，围绕航空数据构建具备后量子迁移方向的安全共享系统，具有明确的研究价值和应用意义。

#### 2.1.3 属性基加密与格密码学的结合方向

在共享控制方面，属性基加密（ABE）能够突破传统“按用户显式授权”的局限，将访问策略直接嵌入密钥封装或密文结构中，从而实现“数据随策略流动”的控制能力。对于航空运维数据场景而言，这种能力具有天然适配性，因为实际业务中经常需要按照航司、部门、岗位、职责、任务场景等多个维度来约束数据访问。

然而，传统 ABE 方案大多基于配对等经典数学问题，其安全基础并不天然具备后量子能力。因此，如果要在“细粒度访问控制”和“后量子迁移潜力”之间同时取得平衡，基于格密码学的属性基加密方案就成为一个更符合项目目标的方向。尤其是 L-ABE 这一类基于格问题的属性基加密方案，兼具策略表达能力和抗量子潜力，因此成为本项目最终目标实现所围绕的核心方向。

### 2.2 研究意义

#### 2.2.1 理论意义

本项目的理论意义主要体现在三个层面。第一，它尝试将格密码学、属性基加密与具体行业数据场景结合起来，不再停留在抽象算法讨论，而是面向航空运行数据这一明确对象研究后量子安全共享体系的可行路径。第二，它探索“数据层对称加密 + 密钥层属性基封装”的分层设计思路，研究如何把复杂密码方案落地到实际 Web 系统中。第三，它通过中期阶段的工程原型验证，为后续从传统过渡方案迁移到真正 L-ABE 封装方案提供系统级参考。

因此，本项目不仅是一个简单的系统开发课题，也不仅是一个纯算法研究课题，而是位于“密码机制研究”与“系统工程集成”之间的交叉型探索。这种研究路径对于后量子密码在行业平台中的实际应用验证具有一定示范意义。

#### 2.2.2 实践意义

从实践角度看，本项目试图为航空 QAR 数据提供一种更安全的处理思路。与传统明文文件管理模式相比，本项目强调文件密文存储、密钥独立封装、访问受控共享以及操作可审计追溯，这一体系更适合高敏感、长周期保存和多主体共享的数据场景。

同时，本项目在中期阶段并未停留在理论设想上，而是已经形成可运行的安全原型系统，完成了认证、审批、审计、文件加密存储、应用层加密传输和飞行数据明文预览等关键链路的验证。这说明项目具有一定工程可行性，也使后续继续推进到 L-ABE 目标方案具有现实基础。

## 三、研究目标与内容

### 3.1 总体目标

本项目的总体目标是构建一个面向航空 QAR 数据的后量子安全存储与可控共享原型系统。其核心思想是采用分层密码架构：在数据层使用 AES-256-GCM 对 QAR 文件主体进行高效加密，在密钥层以 L-ABE 作为最终目标方案对 AES 密钥进行封装，从而实现按属性策略进行数据共享控制。

需要特别说明的是，中期阶段的重点并不是宣称已经完成 L-ABE 的正式工程实现，而是先验证这一最终目标所需要的系统基础，包括：文件密文存储链路是否成立、密钥封装层能否独立存在、服务端和前端能否协同完成加密传输与按需解密、业务流程能否在加密前提下保持可用等。也就是说，中期工作的目标是完成“系统链路验证”和“工程可行性验证”，而最终目标则是在后续阶段将当前过渡性封装方案替换为 L-ABE。

### 3.2 分阶段目标

#### 第一阶段：中期前的系统原型与架构验证

第一阶段的主要目标是完成一个可运行、可演示、可验证的原型系统。在这个阶段中，团队重点完成了前后端基础架构搭建，建立了登录、注册、审批、文件上传下载、反馈与审计等基本业务模块，并将加密能力逐步嵌入文件存储与传输过程之中。

在密码实现方面，第一阶段重点完成了两件事：一是实现 AES-256-GCM 文件加密存储链路，使文件在服务端不再以明文形式保存；二是采用 RSA-OAEP 作为临时过渡封装方案，对 AES 数据密钥进行独立封装，以先验证“数据层与密钥层分离”的混合加密架构。这一阶段还实现了应用层加密传输机制，即客户端按请求动态生成 AES 密钥，并使用服务端公钥进行封装传输，服务端再进行解封装和响应加密返回。

这一阶段的目标不是完成最终密码方案，而是完成最终密码方案所需的系统基础验证。

#### 第二阶段：中期后的 L-ABE 目标推进

第二阶段的核心任务是将当前验证过的系统架构推进到真正面向项目目标的后量子属性基加密体系中。具体来说，这一阶段需要重点完成 L-ABE 的工程实现路线研究、属性策略设计、属性权威建模、属性私钥管理以及性能与安全性评估。

换言之，第二阶段不是从零重新做系统，而是在当前原型和过渡性架构的基础上，把“RSA-OAEP 过渡封装”正式替换为“L-ABE 封装 AES 密钥”的最终方案。

### 3.3 研究内容

#### 3.3.1 混合加密架构设计

本项目首先研究如何在航空运维数据场景下构建一个合理的混合加密架构。由于 QAR 文件通常体量较大、结构复杂，因此不适合直接使用复杂公钥密码对文件主体全文加密，更合理的方式是使用 AES-256-GCM 完成文件正文加密，再将 AES 数据密钥单独交由更高级的封装机制处理。

在中期阶段，这一研究已经转化为工程实现：文件上传后不再以明文直接存储，而是先以 AES-256-GCM 加密，再将密文写入存储字段；AES 密钥则由过渡性方案进行封装保存。后续阶段需要做的，就是把当前封装层替换为 L-ABE，而不是重写整个文件系统。

#### 3.3.2 L-ABE 目标方案研究

本项目的最终密码目标是使用 L-ABE 对 AES 密钥进行属性策略化封装。这一部分研究的重点不是让 L-ABE 直接处理整份 QAR 文件，而是让其承担“密钥层共享控制”的职责，即：谁有权解封装 AES 密钥，谁才有能力解密数据明文。

因此，L-ABE 的研究内容主要包括属性集合设计、访问策略表达、属性权威机制、属性私钥管理以及与当前系统结构的接口衔接问题。中期阶段虽然尚未完成这一部分的正式工程集成，但已经完成了围绕这一目标的系统准备工作。

#### 3.3.3 访问控制与业务模型研究

在系统控制层面，本项目当前已实现基于 RBAC 的角色控制和审批机制，但最终目标并不止于此。项目后续希望将当前“角色权限 + 后台逻辑判断”的方式逐步扩展为“属性集合 + 密钥封装策略”的控制方式。

因此，访问控制研究不仅包括当前的管理员与用户角色划分，也包括后续对航司、部门、岗位、数据类型、职责场景等业务属性的提炼与映射研究。中期阶段已经在档案字段、审批流程和角色模型方面做了前期准备。

#### 3.3.4 工程实现与系统安全研究

本项目并非只关注算法层的安全性，还关注在真实 Web 原型系统中，密码机制如何与认证、审批、审计、预览、展示等业务能力协同工作。中期阶段围绕系统工程实现已经开展了大量工作，包括 Web 安全配置、UTF-8 编码一致性、异常处理、审计日志、飞行数据板块密文条件下的明文展示等。这些内容虽然不属于狭义的密码算法研究，但对“后量子安全共享系统”的实际可用性至关重要。

## 四、技术方案与实现（第一阶段成果）

### 4.1 系统总体架构

#### 4.1.1 系统组成

当前项目已经形成一个可运行的单体 Web 原型系统。后端采用 Spring Boot 4.x，负责认证授权、文件管理、飞行数据解析、管理员后台、反馈与审计等核心能力；前端采用原生 JavaScript 和静态页面实现交互；数据库使用 H2，当前运行配置为内存模式；系统默认运行端口为 `8101`。

从结构上看，系统可以分为前端展示层、后端业务层、数据存储层和密码处理层。其中，前端负责页面展示、表单交互和应用层加密传输配合；后端负责业务流程、权限控制、文件加解密服务和飞行数据解析；数据层负责保存用户、审批、文件记录、日志与档案信息；密码层则负责 AES-GCM 数据加密、RSA 过渡封装以及后端基于 BouncyCastle 的密码实现。

#### 4.1.2 当前架构与最终目标架构关系

当前阶段的密码结构可以概括为：`AES-256-GCM + RSA-OAEP（过渡封装） + RBAC`。最终目标架构则应为：`AES-256-GCM + L-ABE（目标封装） + 属性策略控制`。二者之间并非完全割裂关系，而是“同一整体架构的不同阶段”。

也就是说，当前阶段已经把数据层和密钥层分离，把文件密文存储、密钥独立封装、应用层加密传输和服务端按需解密展示这些关键基础打通；后续真正需要升级的重点，将主要集中在“密钥封装方式”和“访问控制模型”上。

#### 4.1.3 技术选型说明

后端之所以选择 Spring Boot，是因为其在 Java 生态中成熟稳定，便于快速构建 REST 接口、安全模块和数据访问层，也方便与审计、审批、过滤器机制等功能进行集成。前端之所以采用原生 JavaScript，一方面是出于轻量化原型实现考虑，另一方面则便于直接调用浏览器原生的 Web Crypto API，完成应用层加密传输的前端协同验证。

在密码库方面，当前项目已经尽可能把后端的 AES 与 RSA 实现统一到 BouncyCastle 路径上，目的是减少加密实现分散、风格不一致的问题，并为后续进一步扩展复杂密码机制提供更一致的实现基础。

在数据库方面，当前仍使用 H2 内存模式，主要是为了降低原型调试成本，便于中期阶段快速验证与重建环境。但这也意味着项目目前仍偏向原型验证环境，后续还需要补充更稳定的持久化部署方案。

### 4.2 关键技术实现

#### 4.2.1 文件密文存储机制

当前项目已经实现文件存储从明文模式向密文模式的转换。文件上传到服务端后，不再把原始内容直接写入数据库或物理存储，而是先由 AES-256-GCM 进行加密，再保存加密后的内容。对应的 AES 密钥不会以明文形式直接保存，而是交由单独的封装层进行处理。这样的设计，使得文件数据与密钥在系统中分离，有助于降低直接读取存储介质时的明文泄露风险。

在业务侧，这一机制也带来了新的工程问题，例如飞行数据板块不再能直接把存储字段当作明文 xlsx 使用。当前系统已经围绕这一变化完成了适配，支持在预览时由服务端先解密再解析明文数据。

#### 4.2.2 应用层加密传输机制

当前项目实现的不是标准 TLS/HTTPS 协议栈，而是一种面向原型验证的应用层加密传输机制。其核心流程是：客户端先通过握手接口获取服务端公钥，然后针对每一次请求动态生成一个临时 AES 密钥，使用服务端公钥对该临时密钥进行封装，再用该临时 AES 密钥对请求体进行加密。服务端收到请求后，通过过滤器解封装临时 AES 密钥，再解密请求内容；业务处理完成后，服务端再用同一临时 AES 密钥加密响应并返回给客户端。

这一机制的主要意义在于验证“按请求动态生成密钥并进行独立封装传输”的可行性，而不是宣称已经实现了标准 TLS 协议。因此，在报告和答辩中必须明确区分“应用层加密传输”和“标准 HTTPS/TLS”。

#### 4.2.3 用户认证、审批与档案匹配

当前系统在用户准入方面已经形成完整链路。用户注册时，需要提交工号、姓名、身份证后四位、联系方式、航司、职位、部门等信息，系统会将其与人员档案库进行匹配。只有信息匹配成功的用户，才会生成待审批申请，管理员审批通过后才会正式创建账号。

人员档案数据来源于 `person_seed.csv`。在中期调试过程中，团队发现“仅在档案表为空时才导入 CSV”的策略会导致部分工号虽然存在于 CSV，但数据库中并未补齐，从而影响上传时的工号匹配。针对这一问题，当前系统已经改为“启动时补齐缺失工号，并在需要时按工号兜底加载”的策略，从而使档案匹配更加稳定。

#### 4.2.4 审计日志与反馈闭环

审计日志系统负责记录关键请求和操作行为，帮助管理员在事后追踪用户操作、访问路径和系统响应情况。对于一个强调安全与可控共享的系统来说，只有“加密”是不够的，还必须能对访问行为进行留痕和追溯。当前系统已经具备 API 级日志记录能力。

此外，项目还实现了用户反馈与后台处理功能。虽然这一部分不属于狭义密码算法功能，但它能够形成“用户使用 - 问题上报 - 后台处理”的闭环，提高原型系统的完整性和可维护性，也增强了中期答辩中的系统完整度。

#### 4.2.5 飞行数据明文展示能力

飞行数据板块是当前系统中较有代表性的一个工程适配案例。由于底层文件已经改为密文存储，飞行数据板块原先直接把存储字段当作原始 xlsx 解析的方式不再成立。当前项目已经完成修复：在飞行数据预览时，服务端会先调用文件解密逻辑恢复明文，再将明文 xlsx 交给解析组件处理，最终把可视化的明文表格结果返回给页面。

这一能力说明，当前原型已经具备“存储层密文、服务端按需解密、展示层可视明文”的系统基础，这对于后续在更复杂加密方案下维持业务可用性具有重要参考价值。

### 4.3 当前模块完成情况

从整体完成度来看，当前系统已经完成了登录注册、人员档案匹配、管理员审批、文件上传下载、文件统计、反馈处理、飞行数据明文预览、审计日志以及应用层加密传输等关键模块。尚未完成的部分主要集中在最终目标相关能力上，包括 L-ABE 正式封装层、属性权威机制、属性策略表达与完整性能实验。

因此，中期阶段已经具备完整原型系统的主体结构，但距离最终“后量子属性基加密 QAR 系统”的完成状态仍有明确的后续工作。

---

## 五、已完成的工作与阶段成果

### 5.1 原型系统建设成果

当前项目已经形成一套具有实际演示价值的 Web 原型系统。系统支持用户注册、登录、管理员审批、文件上传下载、文件统计查看、飞行数据板块预览、反馈提交和后台日志查询等完整流程。这说明项目不再只是一些零散接口或算法实验，而是具备前后端协同、业务逻辑完整、可实际操作和可直接演示的系统基础。

对于中期答辩而言，这一成果具有重要意义。因为它表明项目不仅“有研究设想”，而且已经形成了可观察、可验证、可说明的阶段性系统成果。

### 5.2 密码与安全能力成果

在密码与安全能力方面，中期阶段已经取得了较为明确的成果。首先，文件上传后已经采用 AES-256-GCM 进行密文存储，这意味着项目已经摆脱了最初的明文文件存储模式。其次，AES 数据密钥已经不再与文件内容混在一起保存，而是采用独立封装方式处理，虽然当前仍为 RSA-OAEP 过渡方案，但这已经完成了对“密钥独立封装”这一最终目标关键结构的验证。

此外，项目还实现了应用层加密传输机制，支持客户端按请求动态生成 AES 密钥并进行公钥封装传输，后端统一进行解封装和数据处理。后端密码实现方面，也已经优先向 BouncyCastle 路径统一。这些工作共同构成了中期阶段的密码与安全成果基础。

### 5.3 关键问题修复成果

项目推进过程中遇到了多项影响系统稳定性和演示效果的问题，其中一些问题本身也具有研究意义。首先，Spring Security 与单页应用异步请求之间的 CSRF 兼容问题，直接影响审批等关键操作的可用性；其次，管理员审批接口权限错误使得后台关键流程无法正常完成；再次，中文乱码问题影响了档案库导入、页面展示和系统整体可用性。

在加密改造过程中，还出现了 `wrapped_key` 字段长度不足导致上传失败、CSV 中存在工号但数据库未补齐导致匹配失败、飞行数据板块因底层改为密文存储而失去明文预览能力等一系列与密码工程集成相关的问题。团队已经逐项完成修复，这些问题的解决不仅提高了系统稳定性，也进一步说明“从算法想法走到系统实现”本身就是本项目研究过程的重要组成部分。

## 六、安全性分析与阶段评估

### 6.1 当前原型已具备的安全能力

从中期阶段成果来看，当前原型已经具备原型级别的完整安全闭环。首先，在数据机密性方面，文件内容已采用 AES-256-GCM 加密存储，不再以明文形式直接落盘。其次，在数据完整性方面，GCM 模式能够同时提供认证加密能力，有助于在数据被篡改时发现异常。

在访问控制方面，系统已经实现基于 Session 和 RBAC 的认证授权模型，同时通过人员档案匹配和管理员审批机制进一步限制系统准入；在行为追溯方面，系统已经具备审计日志记录能力；在传输链路方面，当前原型实现了应用层加密传输，能够在原型阶段降低明文请求直接暴露的风险。

因此，当前系统已经不是单点加密实验，而是包含“数据保护 + 身份控制 + 审批管理 + 审计追溯”的综合性安全原型。

### 6.2 当前阶段的不足

尽管当前原型已经完成多项关键验证，但仍存在明显不足。最核心的一点是：当前密钥封装仍基于 RSA-OAEP 过渡方案，而不是最终目标中的 L-ABE。也就是说，当前系统虽然验证了“密钥独立封装”的系统结构，但尚未真正实现后量子属性基加密。

其次，访问控制仍主要停留在 RBAC 和后台逻辑层面，尚未完成“谁能解密”由属性策略和密码机制直接决定的真正目标。再者，当前数据库仍为 H2 内存模式，系统重启后数据无法长期保留；应用层加密传输也不能等同于标准 HTTPS/TLS。最后，当前阶段还缺少更系统化的性能实验和安全量化分析数据。

### 6.3 阶段性评价

综合来看，当前中期阶段应被评价为“系统链路验证成功，最终目标尚未完成”。它的价值不在于宣称已经得到了完整 L-ABE 系统，而在于已经把未来实现该系统所依赖的关键结构和工程基础做了出来。中期成果的意义在于：项目已经从纯概念阶段进入到可运行、可展示、可继续升级的研究原型阶段。

## 七、存在问题、调整说明与后续计划

### 7.1 当前存在的主要问题

当前项目仍面临几个关键问题。第一，L-ABE 尚未正式完成工程集成，这是项目距离最终课题目标最直接的差距。第二，属性模型、属性权威和属性私钥管理等配套机制尚未建立，当前系统的访问控制仍停留在 RBAC 级别。第三，数据库与部署环境仍偏原型状态，不利于长期演示和稳定积累。第四，当前的实验数据支撑还不充分，尚未形成系统化的性能与安全性分析结论。

这些问题并不意味着当前工作无效，而是说明项目后续仍有较为清晰的深化空间。

### 7.2 路线调整说明

本项目在推进过程中，最重要的调整是将“直接完成 L-ABE 落地”的计划调整为“先构建可运行原型并采用过渡性封装验证系统链路，再逐步替换为 L-ABE”。这一调整的原因在于：L-ABE 本身实现复杂，而项目又需要形成完整系统成果；如果一开始完全压在算法实现上，系统层可能长期缺乏可展示成果。

因此，这一调整不是偏离课题，而是为了提高最终目标达成的可行性。当前所有原型工作，包括文件密文存储、密钥独立封装、应用层加密传输和服务端解密展示，最终都服务于“L-ABE 封装 AES 密钥”的完整目标。

### 7.3 后续计划

近期阶段，项目将继续完善当前原型系统，包括补充测试、稳定核心链路、进一步抽象密钥封装接口、优化数据持久化方案和补充答辩展示材料。中期后阶段，重点将转向 L-ABE 工程路线研究、属性集合设计、策略表达模型与属性权威机制构建。更后续的阶段，则需要开展 L-ABE 与当前过渡方案的性能、安全性和系统可用性对比分析，并逐步完成结题所需的系统整合与材料沉淀。
