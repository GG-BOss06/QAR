## 二、项目背景与意义

### 2.1 研究背景

#### 2.1.1 航空运维数据的安全挑战
- QAR（快速访问记录器）数据是民航飞行品质监控、运行评估、故障分析的重要基础数据，包含飞行高度、速度、姿态、发动机参数、控制面偏转等多类敏感信息。
- 这类数据一方面具有高敏感性和高业务价值，另一方面又常常具有较长保存周期，在事故追溯、运行复盘和长期分析中都需要持续留存。
- QAR 数据并非只被单一主体使用，而是在航空公司、运行控制部门、维修岗位、飞行品质监控岗位等多个主体之间发生共享，这对数据的可控共享提出了明确要求。
- 传统系统通常依赖账号权限或粗粒度 RBAC 模型进行控制，但难以表达某航司 + 某岗位 + 某部门 + 某任务场景这类更细粒度的访问约束。

#### 2.1.2 量子计算对传统密码体系的冲击
- 现有公钥密码体系（如 RSA、ECC）主要依赖经典困难问题，其长期安全性面临量子计算带来的潜在威胁。
- 对于保存周期长、长期保密要求高的航空运行数据而言，即便当前系统未被破解，也存在先截获、后破解的长期风险。
- NIST 已经发布首批后量子密码标准，国际上后量子迁移已成为明确趋势，因此面向未来的航空数据平台有必要提前探索后量子安全架构。

#### 2.1.3 属性基加密与格密码学的结合方向
- 属性基加密（ABE）能够将访问策略嵌入密钥封装或密文结构中，较适合多主体、差异化、按策略共享的航空数据场景。
- 传统 ABE 虽然具备细粒度控制能力，但其经典安全基础难以直接满足后量子方向需求。
- 基于格密码学（如 LWE/RLWE）的 L-ABE 兼顾属性控制与后量子迁移潜力，是本项目最终目标密码方案的重要研究方向。

### 2.2 研究意义

#### 2.2.1 理论意义
- 探索数据层对称加密 + 密钥层属性基封装的混合加密架构在航空运维数据场景中的适用性。
- 研究从中期过渡方案到最终 L-ABE 封装方案的分阶段实现路线，为复杂密码系统的工程化落地提供案例。
- 为后量子密码学在行业数据安全平台中的应用提供具有场景针对性的验证基础。

#### 2.2.2 实践意义
- 为航空 QAR 数据提供一种兼顾存储安全、共享控制和后量子迁移方向的原型平台。
- 相比传统明文存储 + 后台鉴权的方式，构建更强调密文存储、密钥独立封装和可审计共享的系统路径。
- 为后续真正实现L-ABE 封装 AES 密钥的行业级安全应用打下工程基础。

## 三、研究目标与内容

### 3.1 总体目标

构建面向航空 QAR 数据的后量子安全存储与可控共享原型系统，并形成清晰的分层密码架构：

```text
数据层：AES-256-GCM 对称加密，用于高效保护 QAR 文件主体数据
密钥层：最终目标为 L-ABE 封装 AES 密钥，实现按属性策略共享
控制层：当前采用 RBAC + 审批机制，后续向属性策略控制扩展
```

需要强调的是，中期阶段的目标不是宣称已经完成 L-ABE，而是先验证最终目标所需的系统链路、模块边界和工程可行性。

### 3.2 分阶段目标

#### 第一阶段（中期前，当前重点）
1. 完成 Web 原型系统搭建，形成可运行的前后端协同平台。
2. 完成 AES-256-GCM 文件加密存储链路。
3. 采用 RSA-OAEP 作为过渡性密钥封装方案，先验证密钥独立封装的系统架构。
4. 实现应用层加密传输机制，验证按请求动态生成 AES 密钥并进行封装传输的可行性。
5. 实现用户认证、人员档案匹配、管理员审批、审计日志、飞行数据预览等基础业务与安全能力。

#### 第二阶段（中期后，最终目标推进阶段）
1. 调研并确定 L-ABE 的具体工程实现路线。
2. 将当前 RSA-OAEP 过渡封装层替换为 L-ABE 密钥封装层。
3. 设计属性集合、策略表达与业务字段映射模型。
4. 研究属性权威、属性私钥签发与撤销机制。
5. 完成与传统过渡方案的性能、安全性和可用性对比分析。

### 3.3 研究内容

#### 3.3.1 混合加密架构设计
- 研究文件数据使用 AES-256-GCM 加密、密钥单独封装的分层设计。
- 研究中期过渡方案与最终 L-ABE 方案的接口衔接关系。
- 研究文件上传、下载、预览、展示场景下密文存储与按需解密的系统实现。
- 研究服务端与浏览器端协同的应用层加密传输机制。

#### 3.3.2 L-ABE 目标方案研究
- 分析 LWE/RLWE 与属性基加密的结合方向。
- 调研可能的 L-ABE 实现资源与集成方式。
- 研究访问策略、属性集合、属性权威与属性私钥管理问题。
- 明确 L-ABE 在本项目中承担封装 AES 密钥的职责边界。

#### 3.3.3 访问控制与业务模型研究
- 研究用户、管理员、数据所有者、数据使用者等角色的系统关系。
- 研究人员档案字段与未来属性字段之间的映射。
- 研究从当前 RBAC 到后续属性策略控制的渐进式升级路径。
- 研究审批、审计与访问控制之间的协同机制。

#### 3.3.4 工程实现与系统安全研究
- 研究用户认证、CSRF 防护、审计追溯、异常处理等 Web 安全问题。
- 研究飞行数据板块、文件存储板块和后台管理板块在密文条件下的可用性问题。
- 研究中期阶段原型系统如何为最终密码目标提供工程验证基础。

## 四、技术方案与实现（第一阶段成果）

### 4.1 系统总体架构

#### 4.1.1 系统组成
建议从以下几个方面展开：
- 前端：原生 JavaScript、页面交互、加密传输协同。
- 后端：Spring Boot 业务系统、认证授权、文件管理、飞行数据解析、管理员后台。
- 数据层：H2 数据库（当前为内存模式）、文件记录、审计日志、档案数据。
- 密码层：AES-256-GCM 数据加密、RSA-OAEP 过渡封装、BouncyCastle 后端实现。
- 展示层：飞行数据板块服务端解密后预览、文件统计与管理界面。

#### 4.1.2 当前密码架构与最终目标架构对照
建议明确写清：
- 当前阶段：AES-256-GCM + RSA-OAEP（过渡封装）+ RBAC
- 最终阶段：AES-256-GCM + L-ABE（目标封装）+ 属性策略控制

这里要重点强调当前实现是为最终结构验证链路，而非替代最终目标。

#### 4.1.3 技术选型说明
建议逐项解释：
- 为什么后端使用 Spring Boot；
- 为什么当前前端使用 Web Crypto API 配合应用层加密传输；
- 为什么后端优先统一使用 BouncyCastle；
- 为什么当前数据库采用 H2 内存模式；
- 为什么中期先选 RSA-OAEP 作为过渡方案。

### 4.2 关键技术实现

#### 4.2.1 文件密文存储机制
建议说明：
- 文件上传后不再按明文直接存储；
- 文件内容使用 AES-256-GCM 加密；
- AES 数据密钥单独封装保存；
- 当前服务端可在必要场景下进行解密，例如飞行数据板块明文预览。

#### 4.2.2 应用层加密传输机制
需要特别改成符合当前项目现状的表述：
- 当前并非标准 TLS 协议实现，而是应用层加密传输机制；
- 握手阶段由客户端获取服务端公钥；
- 每次请求动态生成 AES 密钥；
- 客户端使用服务端公钥封装临时 AES 密钥；
- 服务端在过滤器中解封装并解密请求，再加密响应返回。

不要继续使用sessionId、固定会话 AES 密钥、完整 TLS 协议这一套旧口径。

#### 4.2.3 用户认证、审批与档案匹配
建议展开说明：
- 登录认证如何完成；
- 用户注册为什么必须经过档案库匹配；
- 管理员审批机制如何限制系统准入；
- `person_seed.csv` 与人员档案导入的关系；
- 导入策略如何从仅初始化调整为补齐缺失工号。

#### 4.2.4 审计日志与反馈闭环
建议说明：
- 审计日志记录哪些内容；
- 审计日志在系统中的作用；
- 反馈系统如何补充业务运维闭环；
- 这些模块为何属于安全原型的重要组成部分。

#### 4.2.5 飞行数据明文展示能力
建议单列一节，说明：
- 文件虽然以密文形式存储；
- 但飞行数据板块需要服务端按需解密后解析 xlsx；
- 当前已实现存储层密文、展示层按需解密的原型能力。
