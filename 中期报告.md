# 大学生创新创业训练计划项目 · 中期报告



一、项目中期总结

1. 项目过程

本项目围绕“面向航空运维数据的安全存储与可控共享”目标持续展开，项目在落地时考虑到实际操作的能力以及项目进展的合理性，对原有计划书进行了一定程度的改动，当前阶段重点取消了直接完成L-ABE算法落地，转而先构建一套可运行、可验证、可扩展的原型系统，验证“对称加密保护数据、非对称机制封装密钥、细粒度权限控制约束访问”的整体路线是否可行。将原本在L-ABE算法先行产出的规划转为前后端先行实现，在此基础上，再推进后续的L-ABE替换与后量子升级。

在技术路线制定阶段，团队采用了“先完成工程闭环，再逐步升级密码方案”的策略。考虑到L-ABE在直接从0实现成本较高，当前系统先以RSA-OAEP作为AES密钥封装的过渡方案，同时统一后端密码实现为BouncyCastle，确保后续更换为L-ABE时，主要调整集中在密钥封装层，而不需要重写业务功能。

现阶段系统已经形成单体Web原型，后端采用Spring Boot 4.x，前端采用原生JavaScript，浏览器侧使用Web Crypto API参与传输密钥协商，服务端加解密逻辑优先使用BouncyCastle实现。系统为本地端口，数据库使用 H2，当前运行配置为内存模式，便于中期阶段快速迭代、反复验证和问题复现。

在系统实现过程中，团队先完成了认证授权、人员档案导入、注册审批、文件上传下载、反馈处理、审计日志等基础业务，再逐步将加密能力嵌入数据传输和文件存储链路。当前上传链路已实现“文件内容经AES-256-GCM加密后再入库，AES密钥经RSA封装保存”，传输链路已实现“每次请求动态生成传输用AES密钥，由客户端用服务端公钥封装后发送”，从而避免在服务端长时间保留明文会话密钥。

在调试阶段，团队围绕系统可用性与安全性进行了多轮修正。已处理的问题包括：Spring Security与SPA交互时的CSRF冲突、管理员审批接口权限异常、中文编码问题、加密实现不统一、文件密钥封装字段长度不足、人员档案CSV与数据库不同步导致的工号匹配失败等。通过这些问题的排查与修复，项目已经从“功能原型”逐步推进到“可实际演示、待持续优化”的阶段。

2. 进展情况、取得的成绩及预期效果

截至目前，项目已完成中期阶段的核心目标，主要成果如下。

（1）完成了可运行的安全原型系统

项目已形成完整的Web原型，具备登录注册、管理员审批、人员档案管理、文件上传下载、反馈提交、审计日志查询等基础功能。系统前后端已打通，能够支撑完整的演示流程与中期检查。

（2）建立了统一的加密技术路线

后端密码操作已优先统一到BouncyCastle方案上，AES-256-GCM 与 RSA-OAEP 的核心实现已集中封装，减少了早期“部分使用JCE、部分使用BouncyCastle”带来的维护成本。当前方案虽然仍是传统公钥封装，但已经为后续过渡到 L-ABE 留出了替换空间。

（3）实现了文件静态加密存储

与项目早期“仅做传输保护”的阶段不同，当前上传文件在服务端已不是明文直接存储，而是以AES-256-GCM密文形式保存；对应AES密钥不以明文保存，而是使用 RSA封装后写入数据库。该改动显著提升了原型系统对“服务器侧数据泄露风险”的防护能力。

（4）实现了应用层加密传输机制

系统当前在应用层实现了加密传输流程：客户端为单次请求动态生成 AES 密钥，使用服务端公钥封装后发送；服务端在过滤器中解封装、解密请求，再对响应进行加密返回。该机制验证了“密钥临时生成、按请求封装传输”的链路设计，为后续替换成 L-ABE 封装机制提供了工程参考。

需要说明的是，当前实现属于应用层加密传输机制，并不等同于标准HTTPS/TLS；标准HTTPS仍计划在后续部署阶段补齐。

（5）完成了严格的人员准入与审批流程

系统已实现基于工号、姓名、身份证后四位、联系方式、航司、职位、部门等字段的人员档案校验机制。用户提交注册申请后，只有管理员审批通过才会正式创建账号。人员档案来源于person_seed.csv。

（6）实现了基于角色的访问控制

系统采用Session + HttpOnly Cookie的认证方式，并基于 RBAC 模型区分ADMIN与 USER两类角色。管理员具备审批、导出、档案管理等能力，普通用户具备查看与下载自身权限范围内数据的能力。系统已完成主要接口级权限控制。

（7）完成了审计与反馈闭环

系统已实现 API 审计日志记录能力，能够记录用户访问路径、请求方式、响应状态等信息；同时实现了用户反馈提交与管理员处理功能，形成了“业务操作 - 问题上报 - 后台处理”的闭环。

（8）解决了一批关键工程问题

中期阶段已实际解决多项影响演示与使用的问题，包括：

- Spring Security 6下 CSRF与SPA请求不兼容问题；
- 管理员审批接口403问题；
- 中文CSV与页面显示乱码问题；
- 启动过程中增量编译和类加载异常问题；
- 文件加密后wrapped_key长度不足导致上传失败的问题；
- 人员档案存在于 CSV 中但数据库未补齐导致上传匹配失败的问题。

总体来看，项目已达到“中期可运行、可演示、可继续扩展”的预期效果，距离“完成L-ABE替换、形成完整后量子安全方案”的最终目标日渐达成。

3. 项目进行中遇到的问题及调整方案

（1）L-ABE工程落地难度高于预期

问题描述：L-ABE的理论与实现复杂度明显高于传统RSA封装方案，尤其在Java生态内缺少成熟、直接可用、易于集成的工程级方案。

调整方案：中期阶段先使用RSA-OAEP作为密钥封装的过渡实现，优先完成系统架构验证与业务闭环，再在后期集中推进L-ABE替换，降低整体开发风险。

（2）加密实现早期不统一

问题描述：项目前期不同模块使用了不同的加密写法，既影响代码一致性，也不利于后期升级。

调整方案：后端统一优先使用BouncyCastle，逐步将AES-GCM、RSA-OAEP等能力集中到工具类和服务层中实现，减少散落在控制器中的重复密码代码。

（3）人员档案导入策略不完善（后续可能再次替换处理）

问题描述：早期CSV 导入逻辑仅在 `person_records` 表为空时执行，一旦数据库中已有部分记录，CSV 中其他工号不会继续补齐，导致前端明明填写了正确工号，上传仍提示档案不存在。

调整方案：已将导入策略改为“启动时补齐缺失工号”，并在上传时增加按工号兜底加载逻辑，确保 CSV 中已有的工号可以及时进入数据库。

（4）文件加密后字段长度不足

问题描述：引入RSA封装后，wrapped_key长度显著增加，数据库字段长度不足导致上传失败。

调整方案：已将wrapped_key调整为大字段存储，以适配当前RSA封装结果，并为后续更长的 L-ABE 封装结果预留空间。

（5）启动与运行稳定性仍需持续优化

问题描述：在频繁调整依赖和实体映射时，出现过spring-boot:run启动失败、增量编译状态异常、类加载失败等问题。

调整方案：当前已通过清理构建、补齐异常处理、增加集成测试等方式降低风险；后续仍需继续优化运行脚本、缓存策略和环境一致性。

（6）当前数据库仍为内存模式

问题描述：项目当前使用H2内存数据库，便于调试，但服务重启后数据不保留，不利于长期演示与持续积累。

调整方案：后期服务器配置后切换到文件模式H2或更稳定的持久化数据库，并同步评估加密数据、审计日志与档案数据的持久化方案。



二、项目后期具体工作计划

后期工作将围绕“算法升级、系统稳定、验证完善”三条主线展开。

近期（1个月）

1. 继续梳理现有RSA封装链路，将密钥封装层进一步抽象，降低后续替换L-ABE的耦合度。
2. 完善传输层、存储层、权限层相关测试，补齐关键链路的自动化验证。
3. 优化人员档案导入、上传下载、管理员审批等核心流程的稳定性与可用性。
4. 评估 H2 文件模式或其他持久化方案，为后续演示和结题提供稳定运行环境。

中远期（2-3个月）

1. 调研并确定L-ABE的最终实现路线，明确是直接Java集成、微服务封装还是第三方库桥接。
2. 在现有RSA封装流程基础上，实现可替换的L-ABE密钥封装模块。
3. 设计属性与策略映射规则，将航司、岗位、部门等业务属性纳入访问控制模型。
4. 补充性能测试、安全性分析与对比实验，评估RSA过渡方案与L-ABE方案在开销和安全目标上的差异。
5. 补充HTTPS、部署配置、运行文档和答辩展示材料，完成结题准备。



三、项目进行过程中遇到的问题及需要的支持

（1）L-ABE 算法实现支持

L-ABE 的工程实现与Java集成方案仍是项目后期的最大难点，希望获得更明确的参考文献、可复用代码或成熟库推荐，以缩短从理论验证到工程实现的周期。

（2）性能测试与实验设计支持

项目后期需要对加密封装方案进行性能评估，希望获得关于测试指标设计、实验方法与对照组设置方面的指导，提升结题数据的可信度。

（3）安全分析与渗透测试支持

目前系统具备原型级安全能力，但在更系统化的安全评估、攻击面分析、漏洞检查方面仍需指导，希望获得相关方法和工具建议。

（4）部署与运行资源支持

若后期切换到更复杂的加密模块或开展性能实验，可能需要更稳定的服务器环境或云资源支持，以便完成连续测试和答辩演示。

（5）项目进度管理支持

项目后期需要同步推进算法研究、系统改造、实验分析与材料撰写，希望在关键节点获得阶段性指导，帮助团队合理把控范围和时间。
