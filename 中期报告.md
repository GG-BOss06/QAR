## 二、项目背景与意义

### 2.1 研究背景

#### 2.1.1 航空运维数据的安全挑战

QAR（Quick Access Recorder，快速访问记录器）数据是航空运行安全监控、飞行品质分析、故障排查和运行评估的重要基础数据，通常包含飞行高度、速度、姿态、发动机工况、控制面偏转等多类敏感信息。这类数据具有两个鲜明特征：一是数据价值高、敏感程度高，关系到飞行安全、运行能力评估以及航班运行分析；二是数据保存周期长，在行业监管和事故追溯中往往需要长期留存。

随着航空运行数字化程度不断提高，QAR 数据的采集、存储、共享与再利用需求不断增长。航空公司、维修单位、运行控制部门、飞行品质监控岗位等不同主体往往都需要在不同权限范围内访问相关数据。然而传统文件共享和后台管理系统通常仅支持粗粒度权限控制，例如按角色或按系统账号区分权限，难以实现“谁在什么条件下可以访问哪类数据”的细粒度管理，这使得数据共享效率和安全控制之间长期存在矛盾。

此外，航空数据具有明显的长期保密特征。当前很多公钥体系虽然在经典计算模型下仍然安全，但从长期保密角度看，已经面临“先截获、后破解”的现实风险。对于需要保存十年以上的航空运行数据而言，即便今天的加密系统没有立即被攻破，只要未来出现足够强的量子计算能力，历史数据也可能面临解密风险。因此，从现在开始探索具备后量子迁移能力的安全架构，具有明显的现实必要性。

#### 2.1.2 量子计算对传统密码体系的冲击

传统互联网和企业系统中大量采用 RSA、ECC 等经典公钥密码机制进行密钥交换、身份认证和数据封装。这些机制的安全性建立在整数分解、离散对数等经典难题之上，而量子计算中的 Shor 算法会对这些难题构成根本性冲击。一旦量子计算进入实用阶段，当前大量依赖 RSA/ECC 的公钥系统将面临整体性安全风险。

近年来，国际上后量子密码迁移趋势已经非常明确。NIST 已发布首批后量子密码标准，全球主流安全体系也都在加快从传统公钥向后量子密码方案迁移。对于航空运维这类对数据长期保存和高安全性要求较高的场景来说，提前验证后量子安全架构的工程可行性，有助于形成更具前瞻性的安全方案。

#### 2.1.3 属性基加密与格密码学的结合趋势

属性基加密（ABE）能够把访问控制策略直接嵌入密钥封装或密文结构中，实现“数据随策略流动”的能力，这与航空运行数据多角色、多部门、多职责共享的业务场景高度契合。与传统 RBAC 相比，ABE 更适合表达类似“某航司 + 某部门 + 某岗位”这类更细粒度的访问条件。

然而，传统 ABE 大多建立在配对等经典困难问题之上，同样无法天然抵抗量子攻击。因此，基于格密码学（如 LWE/RLWE）的 L-ABE 逐渐成为一个具有研究价值的方向。格密码学具有较好的抗量子潜力，而 ABE 又能够满足复杂的访问控制需求，将二者结合有望为航空运行数据提供更长期、更灵活的安全保护机制。

### 2.2 研究意义

#### 2.2.1 理论意义

本项目尝试围绕“对称加密保护数据、非对称/属性机制封装密钥、访问控制与审计共同约束共享行为”这一思路，探索后量子安全架构在航空数据场景下的工程化路径。项目中期阶段虽然尚未直接完成 L-ABE 算法落地，但已经通过构建过渡性的混合加密原型系统，验证了后续向 L-ABE 迁移的整体系统结构是否可行。

从研究角度看，本项目具有以下理论价值：

1. 探索面向航空运维数据的混合加密架构设计思路；
2. 研究“RSA-OAEP 过渡方案到 L-ABE 正式方案”的分阶段演进路线；
3. 为后量子密码在行业数据平台中的工程集成提供一个原型验证案例；
4. 为属性化访问控制与实际业务角色映射提供场景化分析基础。

#### 2.2.2 实践意义

从工程实践角度，本项目的意义主要体现在以下几个方面：

1. 为 QAR 数据安全管理和可控共享提供原型平台；
2. 为航空数据安全平台引入后量子迁移视角，提前验证技术路径；
3. 提供比传统“明文存储 + 粗粒度权限”更安全的文件处理方案；
4. 通过审计、审批、加密存储、应用层加密传输等多种机制，形成更完整的原型级安全闭环。

## 三、研究目标与内容

### 3.1 总体目标

本项目的总体目标是构建一个面向航空 QAR 数据的安全存储与可控共享原型系统，并在架构上预留向后量子密码方案迁移的能力。整体技术路线采用混合加密模式：

```text
数据层：AES-256-GCM 对称加密，用于高效保护文件内容
密钥层：当前采用 RSA-OAEP 作为过渡封装方案，后续替换为 L-ABE
控制层：基于 RBAC 的现有授权模型，后续向属性策略化访问控制扩展
```

也就是说，中期阶段的核心不是直接完成所有后量子算法实现，而是优先验证系统的整体安全链路和工程闭环是否成立。

### 3.2 分阶段目标

#### 第一阶段（中期前，当前已基本完成）

1. 完成 Web 原型系统基础架构搭建；
2. 实现 AES-256-GCM 数据加密能力；
3. 实现以 RSA-OAEP 为过渡方案的 AES 密钥封装；
4. 实现文件上传、下载、预览等核心业务流程；
5. 完成用户认证、审批、审计、反馈等基础安全功能；
6. 打通应用层加密传输链路，验证前后端加解密通信可行性。

#### 第二阶段（中期后，待完成）

1. 调研并实现适用于当前项目的 L-ABE 技术路线；
2. 将现有 RSA 封装层逐步替换为 L-ABE 封装层；
3. 构建属性映射模型与策略表达方式；
4. 实现更细粒度的策略控制能力；
5. 完成性能、安全性与可用性对比分析；
6. 完成持久化、部署优化与结题材料准备。

### 3.3 研究内容

#### 3.3.1 混合加密架构设计

研究如何在一个实际可运行的航空数据平台中，把高效的数据加密与可扩展的密钥封装机制结合起来。中期阶段主要聚焦以下内容：

1. AES-256-GCM 文件加密与认证完整性保护；
2. 文件密钥封装与解封装机制；
3. 应用层请求/响应加密流程；
4. 密钥生命周期的最小化暴露设计；
5. 服务端与浏览器端协同的加密通信方式。

#### 3.3.2 后量子封装路径研究

虽然当前版本仍采用 RSA-OAEP 作为过渡方案，但项目已经把“密钥封装层可替换”作为后续升级目标。研究重点包括：

1. L-ABE 在项目中的接口替换位置；
2. 后续可能采用的开源实现路线；
3. L-ABE 与现有 AES 数据层结合的设计方式；
4. 从角色控制向属性控制过渡的可行性分析。

#### 3.3.3 访问控制与人员准入机制

系统当前已经具备基于 RBAC 的基础权限控制与人员档案匹配机制。研究内容包括：

1. 注册申请与管理员审批流程；
2. 档案库匹配规则设计；
3. 角色权限划分；
4. 属性模型与航空业务字段之间的后续映射思路。

#### 3.3.4 系统安全与工程化实现

项目不只关注算法，还关注工程落地过程中的安全与稳定性问题，包括：

1. 审计日志记录与追溯；
2. CSRF、防越权等 Web 安全处理；
3. 中文编码、文件预览、上传下载等工程细节；
4. 测试、异常处理与演示稳定性保障。

## 四、技术方案与实现（第一阶段成果）

### 4.1 系统架构

#### 4.1.1 系统总体架构

当前项目已形成单体 Web 原型系统。后端采用 Spring Boot 4.x，前端采用原生 JavaScript，数据库使用 H2，当前配置为内存模式，系统默认服务端口为 `8101`。整体结构可概括为：

```text
浏览器前端（HTML/CSS/JavaScript）
        │
        │  应用层加密传输 + 业务请求
        ▼
Spring Boot 服务端
  ├─ 认证授权模块
  ├─ 文件管理模块
  ├─ 飞行数据解析模块
  ├─ 管理员后台模块
  ├─ 审计日志模块
  ├─ 传输层加解密过滤器
  └─ 人员档案导入模块
        │
        ▼
H2 数据库（当前为内存模式）
```

系统现阶段的目标不是做成生产级部署平台，而是构建一个结构清晰、链路完整、可持续演进的研究型原型系统。

#### 4.1.2 混合加密架构

当前系统采用的核心思路是“文件内容与密钥分离处理”：

1. 文件内容使用 AES-256-GCM 进行加密；
2. 文件所用 AES 密钥不明文入库，而是使用 RSA 进行封装；
3. 传输过程中的请求体和响应体支持应用层加密；
4. 服务端通过统一过滤器处理传输层解密与加密。

这一设计为后续替换 L-ABE 提供了明确路径。后续需要替换的重点不是整个系统，而主要是密钥封装层。

#### 4.1.3 渐进式替换思路

本项目在中期阶段作出的关键调整，是将“直接完成 L-ABE”转为“先完成可运行架构验证，再替换封装方案”。因此现阶段与后期目标的关系可以概括为：

```text
当前阶段：
AES-256-GCM + RSA-OAEP(过渡封装) + RBAC

后续阶段：
AES-256-GCM + L-ABE(属性封装) + 属性策略控制
```

这一调整并不是弱化项目目标，而是把复杂的研究目标拆解成更可执行的工程阶段。

### 4.2 关键技术实现

#### 4.2.1 AES-256-GCM 数据加密

系统当前对文件数据采用 AES-256-GCM 算法进行加密。选择 GCM 模式的主要原因在于其兼顾了机密性和完整性校验，能够在加密同时提供认证标签，降低数据被篡改而不被发现的风险。

在具体实现上，服务端通过统一工具类完成 AES 密钥生成、随机 IV 生成、加密和解密逻辑。文件上传后不再以明文形式直接存储，而是先进行 AES-256-GCM 加密，再将密文写入存储字段中。这样一来，即使数据库数据被直接读取，也无法在没有密钥的情况下恢复出原始明文。

#### 4.2.2 RSA 封装与服务端密钥对管理

中期阶段尚未完成 L-ABE 算法落地，因此系统当前使用 RSA-OAEP 作为 AES 密钥封装的过渡方案。服务端专门实现了独立的密钥对服务，用于管理服务端 RSA 公私钥对，并支持以下能力：

1. 启动时生成或加载服务端 RSA 密钥对；
2. 提供服务端公钥给客户端用于临时传输密钥封装；
3. 对文件 AES 密钥进行封装并保存；
4. 在需要时进行密钥解封装。

这种设计使得 AES 密钥不需要以明文长期暴露，同时也为后续替换到 L-ABE 提供了较清晰的接口位置。

#### 4.2.3 应用层加密传输

系统并未直接把“前后端请求是否走 HTTPS”作为唯一安全路径，而是额外实现了应用层加密机制。其基本流程为：

1. 客户端通过握手接口获得服务端公钥；
2. 客户端为单次请求动态生成 AES 密钥；
3. 客户端使用服务端公钥封装该 AES 密钥；
4. 请求体使用 AES-GCM 加密后发送；
5. 服务端在过滤器中解封装密钥并解密请求；
6. 响应返回前再用同一临时 AES 密钥加密。

这种做法的直接收益是：即使在原型阶段尚未完成标准 HTTPS 部署，也能够验证“动态生成临时密钥并按请求封装传输”的系统链路，为后续进一步升级到后量子密钥封装提供工程基础。

需要说明的是，这一机制属于应用层加密传输，并不等同于浏览器和服务器之间的标准 TLS/HTTPS。标准 HTTPS 仍属于后期部署优化内容。

#### 4.2.4 用户认证与角色授权

系统当前采用 Session + HttpOnly Cookie 方式实现登录态管理，并基于 Spring Security 完成接口级访问控制。权限模型目前采用 RBAC，主要区分两类角色：

1. `ADMIN`：具备管理员后台、档案管理、审批、导出、全局查询等权限；
2. `USER`：具备工作台、上传下载、查看本人相关数据等权限。

虽然当前控制粒度主要停留在角色层，但该模型已经满足中期阶段的业务原型需求，并可在后续扩展为属性映射控制。

#### 4.2.5 人员档案导入与准入审批

为避免任意注册导致系统身份失真，项目设计了“档案匹配 + 审批通过”双重准入机制。人员档案来自 `person_seed.csv`，注册时会校验工号、姓名、身份证后四位、联系方式、航司、职位、部门等字段；只有档案匹配成功且管理员审批通过，系统才会正式创建用户账号。

在中期调试过程中，团队进一步优化了档案导入机制：原先仅在表为空时整批导入 CSV，后来改为“启动时补齐缺失工号 + 上传时按工号兜底加载”，从而提升了系统对真实测试流程的稳定支持能力。

#### 4.2.6 审计日志与反馈闭环

为提高系统可追溯性，项目实现了全局 API 审计日志记录。系统能够记录访问时间、请求路径、请求方法、响应状态以及用户身份等关键信息，管理员可以在后台进行查询和追踪。同时，系统还实现了用户反馈与后台处理功能，形成从问题上报到处理闭环的原型能力。

#### 4.2.7 飞行数据板块的服务端明文预览

随着文件改为密文存储，项目还同步修正了飞行数据预览链路：服务端在解析 Excel 文件前先进行解密，再将得到的明文数据交由解析组件处理，从而保证飞行数据板块仍能展示可视化的明文内容。这一修复说明当前系统已经具备“存储层密文、展示层按需解密”的基本能力。

## 五、已完成的工作

### 5.1 业务原型系统建设

中期阶段已经完成完整的 Web 原型系统搭建，前端页面和后端接口能够支持以下业务链路：

1. 登录、注册、注销、查看当前登录状态；
2. 管理员审批注册申请；
3. 用户上传文件并查看文件列表与统计信息；
4. 管理员查看人员档案、全局文件信息与审计日志；
5. 飞行数据 Excel 文件的服务端预览与展示；
6. 用户反馈提交与处理。

这说明系统已经从“单点功能验证”发展到“具备完整演示流程的原型平台”。

### 5.2 安全能力建设

在安全能力方面，中期阶段已经完成以下工作：

1. 建立 AES-256-GCM 文件加密存储机制；
2. 建立 RSA 过渡封装机制，避免文件 AES 密钥明文保存；
3. 建立应用层传输加密机制；
4. 建立基于 Spring Security 的认证与权限控制；
5. 建立全局审计日志机制；
6. 建立注册审批与人员档案校验机制。

这些工作共同构成了当前系统的“原型级安全闭环”。

### 5.3 关键工程问题修复

在项目推进过程中，团队并非只实现了功能，更解决了一批影响系统稳定性和可演示性的关键问题，主要包括：

1. 修复了 Spring Security 与 SPA 异步请求之间的 CSRF 兼容问题；
2. 修复了管理员审批接口的权限异常问题；
3. 修复了中文 CSV 导入、数据库读取和页面显示中的乱码问题；
4. 修复了上传链路中 `wrapped_key` 字段长度不足的问题；
5. 修复了 CSV 中存在工号但数据库未补齐时上传匹配失败的问题；
6. 修复了飞行数据板块因底层存储转为密文而导致的明文预览失败问题；
7. 优化了启动与编译过程中出现的类加载异常和增量编译状态问题。

这些问题的修复直接提升了中期答辩和后续开发的稳定性。

### 5.4 加密技术路线统一

项目前期曾存在加密实现方式不够统一的问题，部分逻辑使用标准 JCE，部分逻辑使用不同风格的加密代码。当前已按照项目规范优先统一为基于 BouncyCastle 的后端密码实现，并将 AES 和 RSA 相关能力尽量集中到工具类与服务层，减少了控制器中散落的密码代码。这一工作为后续 L-ABE 替换打下了更好的结构基础。

### 5.5 可持续演进的架构基础

虽然当前还未完成 L-ABE，但从系统结构上看，项目已经完成了两个关键前提：

1. 文件数据层与密钥封装层已经分离；
2. 传输加密与业务逻辑已经通过过滤器机制解耦。

这意味着未来若替换成新的密钥封装方案，重点修改可以集中在封装与解封装层，而不必重构整个文件管理和业务系统。

## 六、安全性分析

### 6.1 当前阶段已具备的安全能力

从中期阶段成果看，系统在以下方面已经具备基础安全能力：

| 安全维度 | 面临威胁 | 当前措施 | 当前评价 |
|---------|----------|----------|----------|
| 数据机密性 | 文件被直接读取 | AES-256-GCM 密文存储 | 已具备基本保护 |
| 数据完整性 | 文件内容被篡改 | GCM 认证标签 | 已具备校验能力 |
| 密钥保护 | AES 密钥泄露 | RSA 过渡封装 | 优于明文保存 |
| 传输安全 | 请求内容被直接窃听 | 应用层 AES-GCM 加密 | 原型阶段有效 |
| 身份认证 | 未登录访问 | Session + HttpOnly Cookie | 已实现 |
| 访问控制 | 越权访问 | 基于角色的接口控制 | 已实现 |
| 操作追溯 | 事后无法定位责任 | 审计日志记录 | 已实现 |
| 准入安全 | 非授权人员注册 | 档案匹配 + 管理员审批 | 已实现 |

### 6.2 当前阶段的局限性

虽然系统已经具备较完整的原型级安全能力，但从“后量子安全”和“生产级部署”角度看，仍有以下不足：

1. 当前密钥封装仍基于 RSA，不具备真正的抗量子能力；
2. 当前访问控制仍基于角色，不是属性基细粒度控制；
3. 当前数据库仍为 H2 内存模式，重启后数据无法长期保留；
4. 当前传输保护属于应用层加密机制，尚未部署标准 HTTPS；
5. 当前系统更多强调原型验证，尚未进行完整的系统化安全评估。

### 6.3 后续安全增强方向

后期若完成 L-ABE 集成，系统的安全能力将得到进一步增强，主要体现在：

1. 密钥封装具备后量子迁移方向；
2. 访问控制从 RBAC 升级为属性策略控制；
3. 数据共享可以表达更复杂的业务规则；
4. 系统能更好地兼顾数据安全与多角色共享需求。

## 七、存在的问题与后续计划

### 7.1 当前存在的主要问题

#### 7.1.1 L-ABE 尚未完成工程集成

这是当前项目距离原始课题目标仍然存在的最大差距。虽然系统结构已经为后续替换预留了位置，但 L-ABE 的具体实现、封装、性能与工程适配仍待完成。

#### 7.1.2 访问控制粒度仍较粗

现阶段系统仍以角色权限为主，尚未达到“基于航司、岗位、部门、数据类型”等多属性条件进行细粒度控制的目标。

#### 7.1.3 数据持久化能力仍不足

当前数据库配置为 H2 内存模式，适合原型调试，但不利于长期演示、稳定测试与后续研究积累。

#### 7.1.4 部署与运行环境仍偏开发态

目前系统更适合本地运行与课堂演示，尚未形成完善的部署说明、运行监控、异常恢复和标准 HTTPS 支持。

#### 7.1.5 性能分析与实验数据仍不充分

中期阶段重点在于打通链路与修复问题，因此当前尚缺乏系统化的性能对比实验、安全评估和定量分析结论。

### 7.2 后续工作计划

#### 近期计划（1-2个月）

1. 继续梳理现有 RSA 过渡封装实现，进一步抽象封装接口；
2. 补齐关键业务链路的自动化测试与异常处理；
3. 优化人员档案导入、飞行数据展示、管理员流程等稳定性问题；
4. 评估将 H2 切换为文件模式或其他持久化方案的可行性；
5. 完善中期答辩与阶段总结材料。

#### 中期计划（2-4个月）

1. 调研并确定 L-ABE 的具体实现路线；
2. 开始进行 L-ABE 密钥封装层的原型集成；
3. 设计属性与业务字段的映射关系；
4. 研究访问策略表达方式；
5. 逐步把现有 RBAC 与后续属性控制模型进行衔接。

#### 远期计划（4-6个月）

1. 完成 L-ABE 与现有系统的接口整合；
2. 开展 RSA 过渡方案与 L-ABE 方案的性能对比测试；
3. 完成安全性分析、系统优化与部署加固；
4. 补充标准 HTTPS 和更稳定的运行环境；
5. 完成结题报告、展示材料和项目总结。
