# 安全需求分析报告

## 一、文档目的

本报告围绕项目所面向的业务场景、关键保护对象、参与主体、安全需求与密码需求展开分析，用以说明为什么最终方案必须走向L-ABE封装AES密钥的结构。

## 二、业务场景分析

本项目主要面向航空运维数据场景，核心对象为 QAR 数据。QAR 数据在飞行品质监控、故障定位、运行评估等场景下具有重要价值，往往由多个主体在不同职责范围内共享和使用。

相关主体包括但不限于：

1. 航司运行部门；
2. 飞行品质监控岗位；
3. 维修工程师；
4. 管理员或数据平台运维人员；
5. 其他依法依规授权的数据使用者。

这些主体并非都应拥有相同的数据访问能力，因此系统必须支持差异化、细粒度、可审计的数据共享机制。

## 三、保护对象识别

### 3.1 核心数据资产

1. QAR 原始文件和解析后的飞行数据；
2. 文件所使用的 AES 数据密钥；
3. 用户身份信息、档案信息与审批信息；
4. 属性信息与访问策略配置；
5. 审计日志和反馈记录。

### 3.2 敏感性分析

其中最敏感的对象主要是：

1. QAR 数据明文；
2. 可解密 QAR 数据的 AES 密钥；
3. 决定谁能访问数据的属性密钥或策略配置。

## 四、安全需求分析

### 4.1 数据机密性需求

系统必须确保：

1. 未被授权的用户无法获得 QAR 数据明文；
2. 即便存储介质被直接读取，也不能直接获得原始数据；
3. 数据在传输和展示过程中只在必要范围内暴露明文。

### 4.2 数据完整性需求

系统必须确保：

1. 文件内容在存储和传输中被篡改时能够被发现；
2. 明文解析与业务处理所依赖的数据未遭静默破坏；
3. 展示结果与真实文件内容一致。

### 4.3 可控共享需求

系统不应仅实现登录后可见这种粗粒度控制，而应支持：

1. 不同岗位、部门、航司之间的数据访问差异；
2. 数据所有者或管理者可按业务需求定义访问策略；
3. 数据共享范围不再仅依赖后台管理员手工判断。

### 4.4 长期保密需求

考虑到 QAR 数据具有较强的长期留存需求，系统需要具备面向后量子密码迁移的技术准备，避免未来传统公钥体系失效时导致历史数据整体失守。

### 4.5 可追溯需求

一旦发生异常访问、策略误配或泄露争议，系统必须具备追踪能力，包括：

1. 谁访问过；
2. 何时访问；
3. 访问了什么；
4. 是否符合审批与权限流程。

### 4.6 工程可用性需求

安全系统必须可运行、可演示、可维护。若密码方案虽然理论先进但工程上不可落地，则无法支撑项目目标。因此需要兼顾：

1. 前后端协同可实现；
2. 文件上传、下载、预览流程可运转；
3. 管理流程与安全流程不冲突；
4. 后期能够逐步升级而非完全推倒重来。

## 五、密码需求分析

基于上述安全需求，系统在密码层面可归纳出如下要求：

1. 数据层需要一种高效、稳定、适合大文件的加密算法；
2. 密钥层需要一种支持访问策略控制的封装机制；
3. 整体系统需要具备向后量子密码迁移的能力；
4. 密钥与数据本体需要相互分离管理；
5. 密钥使用过程需要与用户属性和策略绑定。

从这些需求出发，最终最合理的密码设计就是：

- 使用 AES-256-GCM 加密 QAR 数据；
- 使用 L-ABE 封装 AES 数据密钥；
- 使用属性私钥满足策略后获得数据访问能力。

## 六、当前原型对需求的满足情况

### 6.1 已满足的需求

当前原型已经基本满足：

1. 文件密文存储需求；
2. 应用层传输加密需求；
3. 用户准入与角色控制需求；
4. 审计追溯需求；
5. 前后端链路可运行需求。

### 6.2 尚未完全满足的需求

当前尚未完全满足：

1. 后量子密钥封装需求；
2. 属性策略化共享需求；
3. 属性权威与属性密钥管理需求；
4. 长期稳定部署与持久化需求。

## 七、需求分析结论

综合业务场景与安全目标可以得出如下结论：

1. 本项目不适合做成简单的登录鉴权 + 文件上传系统；
2. 仅靠 RBAC 不能满足最终的可控共享需求；
3. 仅靠传统公钥封装难以回应后量子方向要求；
4. AES-256-GCM + L-ABE是与业务场景最匹配的最终密码结构；
5. 当前原型工作已经在支撑该结构，但仍需后续完成 L-ABE 集成和属性策略体系建设。
