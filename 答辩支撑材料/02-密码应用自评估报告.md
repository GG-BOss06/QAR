# 密码应用自评估报告

## 一、评估目的

本报告从项目最终目标和当前中期实现两个层面，对密码应用方案进行系统自评估。评估重点在于分析该方案是否具有完整的密码体系、是否与题目目标一致，以及当前过渡实现与最终目标之间的关系是否清晰。

本报告的主要目的包括：

1. 说明本项目拟采用的密码应用体系是否完整；
2. 评估当前方案与最终目标方案之间的关系；
3. 梳理系统的安全目标、密码边界和风险点；
4. 为后续阶段的方案论证、风险说明和材料使用提供自评估依据。

## 二、系统密码应用场景概述

本项目面向航空运维 QAR 数据场景。此类数据具有以下特点：

1. 数据敏感度高，涉及飞行运行细节与安全分析信息；
2. 数据保存周期长，存在长期保密需求；
3. 数据共享主体多，存在跨角色、跨部门访问场景；
4. 传统账号权限模型难以表达复杂业务共享规则。

因此，项目最终密码应用目标并不是单纯保护传输链路或数据库文件，而是希望形成一套贯穿数据加密、密钥分发、策略控制和审计追溯的完整安全体系。

## 三、密码应用目标与安全目标

### 3.1 密码应用目标

本项目密码应用的最终目标是：

1. 对 QAR 文件数据进行高强度加密存储；
2. 对文件所使用的 AES 密钥进行策略化封装；
3. 仅允许属性满足访问策略的主体解封装并访问明文；
4. 兼顾工程可实现性、业务可用性与后量子迁移潜力。

### 3.2 核心安全目标

从系统安全角度，本项目主要追求以下目标：

1. 机密性：非授权主体无法读取 QAR 明文数据；
2. 完整性：文件在存储和传输中被篡改时能够被发现；
3. 可控共享：数据不是对所有合法登录用户开放，而是应符合特定属性策略；
4. 长期保密：方案应具备面向后量子密码迁移的潜力；
5. 可审计性：对关键访问和操作行为可追踪；
6. 工程可落地性：不能只停留在理论方案，必须能与实际 Web 系统结合。

## 四、目标密码体系自评估

### 4.1 数据层：AES-256-GCM

自评结论：适合作为最终数据层加密方案。

评估理由：

1. AES 是成熟标准算法，工程实现稳定；
2. 256 位密钥长度具备较高安全强度；
3. GCM 模式适合文件场景，可同时提供机密性和完整性验证；
4. 相对于公钥密码，AES 更适合大文件数据主体加密。

风险与注意事项：

1. IV 生成必须随机且不重复；
2. 密钥生命周期管理必须严格受控；
3. AES 只解决数据层问题，无法独立完成访问控制与共享策略。

### 4.2 密钥层：L-ABE 封装 AES 密钥

自评结论：作为最终目标方案合理，且与项目主题高度一致。

评估理由：

1. L-ABE 同时具备属性策略控制与后量子方向潜力；
2. 能够把谁有资格访问数据嵌入密钥封装机制，而不是仅依赖后台权限判断；
3. 与 QAR 数据多主体共享、长期保存、高敏感的场景相匹配；
4. 可以自然承接项目题目中的后量子可控共享属性基加密要求。

风险与挑战：

1. Java 生态中的成熟工程实现资源有限；
2. 策略表达、属性权威、密钥撤销等配套设计复杂；
3. 性能开销、密文长度与系统集成复杂度可能高于传统方案。

需要补充说明的是，NIST当前重点推进的后量子算法如ML-KEM、ML-DSA、SLH-DSA、FN-DSA和HQC，分别主要服务于KEM封装或签名场景。若项目后续需要把当前RSA-OAEP过渡封装升级为标准化后量子点对点封装方案，ML-KEM会是更优先考虑的候选；若项目需要补充后量子签名能力，ML-DSA通常是更平衡的选择。但这些算法并不具备属性策略表达能力，因此它们可以作为外围后量子能力补充，而不能替代L-ABE在本项目中的核心作用。

### 4.3 控制层：从 RBAC 向属性策略控制过渡

自评结论：当前 RBAC 适合作为中期过渡控制模型，最终仍需升级。

评估理由：

1. 当前 RBAC 适合快速完成原型系统基本业务；
2. 有助于先验证用户准入、管理员审批、日志审计等链路；
3. 但 RBAC 无法替代 ABE 的密码级共享控制，最终必须向属性模型演进。

### 4.4 审计层与配套机制

自评结论：构成密码应用方案完整性的必要部分。

评估理由：

1. 密码保护解决能否看见数据的问题；
2. 审计和审批机制解决谁访问过、是否合规、如何追责的问题；
3. 对于答辩展示而言，这部分能体现项目不是孤立算法验证，而是完整系统研究。

## 五、当前中期原型的自评估

### 5.1 已具备的能力

当前原型虽然尚未完成 L-ABE，但已经验证了以下重要环节：

1. QAR 文件可通过 AES-256-GCM 进行密文存储；
2. 文件 AES 密钥可独立于文件内容进行封装；
3. 前后端之间可以实现按请求动态生成临时 AES 密钥并封装传输；
4. 服务端可以按需解密并提供飞行数据板块的可视明文解析；
5. 用户认证、审批、审计等外围机制已打通。

### 5.2 与最终目标的差距

当前原型距离最终目标仍存在以下差距：

1. 密钥封装仍基于 RSA-OAEP，而非 L-ABE；
2. 访问控制仍主要依赖 RBAC，而非属性策略；
3. 尚未构建属性权威与属性密钥管理体系；
4. 缺少针对 L-ABE 方案的性能实验与安全分析数据。

### 5.3 中期原型的合理性判断

自评结论：合理且必要。

原因如下：

1. 若一开始直接实现 L-ABE，工程风险过高，可能导致整个系统长期停留在算法验证层面；
2. 当前原型先验证混合加密架构、文件密文存储、封装传输和展示链路，是推进最终目标的合理步骤；
3. 过渡原型不是偏离课题，而是在为最终 L-ABE 集成做系统级准备。

## 六、项目密码应用存在的风险点

1. 算法实现风险：L-ABE 方案的具体技术路线仍需进一步明确；
2. 工程集成风险：L-ABE 与现有 Java Web 系统集成复杂度较高；
3. 性能风险：策略复杂度提升后，封装/解封装开销可能显著增加；
4. 密钥管理风险：属性私钥签发、撤销、更新流程尚未正式设计完成；
5. 部署风险：当前系统仍为原型环境，生产级密码部署策略尚未完善。

## 七、综合自评结论

综合评估认为，本项目的密码应用目标具有明确的研究指向和工程价值，最终方案L-ABE 封装 AES 密钥与项目题目、行业场景和后量子迁移需求高度一致。从密码体系设计角度看，该方案结构完整、目标清晰、层次分明。

当前中期阶段虽然尚未完成最终目标实现，但已通过 RSA 过渡封装方案验证了混合加密架构、密钥独立封装、应用层加密传输和文件密文存储的可行性。因此，当前阶段工作可以视为最终密码应用系统的前置验证，而不是偏离项目主题的临时实现。

本项目后续应继续围绕 L-ABE 封装 AES 密钥这一核心目标推进研究与工程集成，尽快完成属性模型、封装算法、性能评估和系统替换工作。
